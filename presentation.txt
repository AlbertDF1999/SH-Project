Slide 1: Title
"ERC-4337 Account Abstraction Smart Wallet"

Hi everyone. Today I'm going to walk you through a project I built — a production-level smart contract wallet that implements ERC-4337, the Account Abstraction standard on Ethereum. I'll explain the problem it solves, how it works under the hood, and the advanced features I implemented. The entire system is deployed and verified on Sepolia testnet.


Slide 2: The Problem — Why We Need Account Abstraction
"The limitations of traditional Ethereum wallets"

To understand why this project exists, we need to understand the problem with how Ethereum wallets work today.
Every Ethereum wallet you've used — MetaMask, Coinbase Wallet, Trust Wallet — is what's called an Externally Owned Account, or EOA. It's just a private key and an address. And EOAs have serious limitations.
First, you need ETH to do anything. You can't interact with a dApp if you don't have ETH for gas, even if you're just approving a token. Second, if you lose your private key, your funds are gone forever. There's no recovery mechanism. Third, each transaction requires a separate signature and submission — if you want to approve a token and then swap it, that's two transactions, two gas payments, two confirmations.
And finally, there's no way to delegate permissions. You either have full control of the wallet or no control at all. There's no middle ground like "let this gaming app make moves for the next two hours but nothing else."
ERC-4337 solves all of these problems by turning the wallet itself into a smart contract.


Slide 3: What is ERC-4337?
"Account Abstraction — wallets as smart contracts"

ERC-4337 is a standard that introduces Account Abstraction to Ethereum without requiring any changes to the protocol itself. Instead of sending regular Ethereum transactions, users create objects called UserOperations — or UserOps — which describe what they want to do.
These UserOps are collected by actors called bundlers, who package them into a regular Ethereum transaction and submit them to a special singleton contract called the EntryPoint. The EntryPoint then calls each user's smart contract wallet to validate the signature and execute the operation.
The key insight is that the validation logic lives inside the smart contract. This means each wallet can define its own rules for what constitutes a valid signature, how gas is paid, and what operations are allowed. That's where the real power comes in.


Slide 4: Architecture Overview
"Four contracts working together"

Let me walk you through the architecture of my implementation. There are four main components.
First, the EntryPoint — this is the canonical ERC-4337 v0.7 contract deployed at the same address on every EVM chain. It orchestrates the entire UserOp lifecycle: validation, gas accounting, execution, and bundler compensation. I didn't write this — it's the standard singleton that every ERC-4337 wallet interacts with.
Second, the MainAccount — this is the core wallet contract I wrote. It implements the IAccount interface, which requires a validateUserOp function. This contract contains all the wallet logic: executing transactions, validating signatures, managing session keys, and handling recovery.
Third, the MainAccountFactory — this is the factory contract that creates new wallets using the CREATE2 opcode. It deploys one MainAccount as a shared implementation, then creates lightweight proxies for each individual user's wallet.
Fourth, the ERC1967 Proxy — each user's wallet is actually a tiny proxy contract, only 212 bytes, that delegates all calls to the shared implementation via delegatecall. This means every proxy has its own storage — its own owner, session keys, and guardian — but shares the same code. This saves massive gas compared to deploying the full contract for every user.


Slide 5: The UserOperation Flow
"What happens when you send a transaction"

Let me trace through exactly what happens when a user sends a UserOperation through my wallet. I'll use real numbers from a transaction I executed on Sepolia.
Step one: the bundler — in my testing, my own wallet acting as the bundler — calls handleOps on the EntryPoint. This is a regular Ethereum transaction that the bundler pays gas for. In my transaction, this cost 128,016 gas at an effective gas price of 0.001 gwei.
Step two: the EntryPoint calls validateUserOp on my proxy wallet. The proxy forwards it via delegatecall to the implementation. Inside, my contract recovers the signer's address using ecrecover on the EIP-191 signed message hash. It checks if the signer is the owner or a valid session key. This validation used 42,064 gas.
Step three: during validation, the wallet pays a prefund to the EntryPoint. This is the worst-case gas deposit, calculated as (verificationGasLimit + callGasLimit + preVerificationGas) × maxFeePerGas. In my case, that was 1,100,000,000,000 wei — about 0.0000011 ETH.
Step four: the EntryPoint calls execute on my wallet, which transfers 0.001 ETH to my destination address. The execution phase used 8,694 gas.
Step five: the EntryPoint calculates the actual gas cost — 226,242 gas total — and compensates the bundler from the wallet's deposit. The unused portion stays credited to the wallet for future operations.


Slide 6: Feature 1 — Batch Execution
"Multiple transactions in a single operation"

The first advanced feature is batch execution. The executeBatch function takes three parallel arrays — destinations, values, and calldata — and executes them atomically in a single UserOperation.
Why does this matter? Consider a common DeFi workflow: you want to approve USDC on a DEX and then swap it for ETH. With a traditional EOA, that's two separate transactions — two signatures, two gas payments, two rounds of waiting for confirmations. With executeBatch, both operations happen in one UserOp — one signature, one validation, one gas payment.
The execution is atomic, meaning if any call in the batch fails, the entire batch reverts. There's no partial state where you've approved tokens but the swap failed, leaving an open approval for someone to exploit.
Under the hood, it loops through the arrays and makes low-level call operations to each destination, checking the success boolean after each one. The requireFromEntryPointOrOwner modifier ensures only the EntryPoint or the wallet owner can invoke it.


Slide 7: Feature 2 — Session Keys
"Delegated permissions without sharing your private key"

Session keys are probably the most powerful feature in this wallet. They solve a fundamental problem: how do you let a third party perform actions on your behalf without giving them full control?
Each session key is constrained by four parameters. First, a time window — the key has a validAfter timestamp and a validUntil timestamp, so it automatically expires. Second, a target contract — you can restrict the key to only interact with a specific contract address, or set it to address(0) for any contract. Third, a function selector — you can restrict it to only call a specific function, like transfer, or set it to bytes4(0) for any function. Fourth, an active status — the owner can revoke any session key instantly.
The _validateSignature function handles this. When a UserOp arrives, it recovers the signer and first checks if it's the owner — if so, immediate success with no restrictions. If not, it looks up the signer in the session keys mapping and validates all four constraints. To check the target and function, it parses the UserOp's ABI-encoded calldata — the target address sits at bytes 4 through 36 of the execute calldata, and the nested function selector sits at bytes 132 through 136.
A real-world example: you're playing a blockchain game. You create a session key that can only call makeMove() on the game contract for the next two hours. The game can submit moves on your behalf without you signing every single action. And if that session key gets compromised, the attacker can only make game moves — they can't drain your wallet or interact with any other contract.


Slide 8: Feature 3 — Social Recovery
"Recover your wallet if you lose your private key"

The third feature is guardian-based social recovery. This solves the biggest problem in crypto — if you lose your private key, your funds are gone forever. With social recovery, a trusted guardian can transfer ownership to a new address.
Here's how the flow works. The owner sets a guardian using setGuardian — this could be a trusted friend, family member, or a separate hardware wallet. Only the owner can set or change the guardian.
If the owner loses access, the guardian calls initiateRecovery with the proposed new owner address. This starts a mandatory 2-day waiting period. After the 2 days pass, the guardian calls executeRecovery to finalize the ownership transfer.
The 2-day delay is the critical security element. Without it, a compromised guardian could instantly steal the wallet. With it, the owner has time to notice — through event monitoring or notifications — and call cancelRecovery to stop a malicious attempt.
There's also an important subtlety in the implementation: every new initiateRecovery call resets the timer. This prevents an attack where the guardian initiates recovery with a benign address to run out the clock, then swaps in a malicious address at the last second. Each new proposed owner gets a fresh 2-day countdown.


Slide 9: Feature 4 — Factory Pattern with CREATE2
"Deterministic wallet addresses and gas-efficient deployment"

The fourth feature is the factory deployment pattern. Instead of deploying a full 11,708-byte MainAccount contract for every user, the factory uses the ERC1967 proxy pattern combined with CREATE2.
Here's how it works. When the factory is deployed, its constructor creates a single MainAccount as the shared implementation and calls _disableInitializers to permanently lock it — preventing anyone from calling initialize on the implementation directly and potentially compromising every proxy that delegates to it.
When createAccount is called, the factory deploys a lightweight ERC1967 proxy — only 212 bytes — using CREATE2 with the provided salt. The proxy's constructor atomically stores the implementation address and calls initialize(owner) via delegatecall, which sets the owner in the proxy's own storage.
CREATE2 is what makes this powerful. The address is deterministically computed from four inputs: a fixed prefix, the factory address, the salt, and the hash of the bytecode. This means you can compute a wallet's address before it exists on-chain. A user can share their wallet address and receive funds before their wallet is even deployed.
In the full ERC-4337 flow, the user's first UserOperation includes initCode — the factory address plus the createAccount calldata. The EntryPoint calls the factory to deploy the wallet, then validates the signature and executes the operation, all in one transaction. The user never manually deploys anything.
The function is also idempotent — if you call createAccount with the same owner and salt twice, it checks addr.code.length, sees the proxy already exists, and simply returns it without reverting. This is essential because the EntryPoint might call it redundantly.


Slide 10: The Proxy Pattern Deep Dive
"One implementation, many wallets"

I want to spend a moment on why the proxy pattern matters, because it's central to the architecture.
When you call a function on a proxy, say execute, the proxy doesn't have that function in its own code. Instead, it catches the call with its fallback function and forwards it to the implementation using delegatecall. The critical property of delegatecall is that it runs the implementation's code but uses the proxy's storage.
So when the implementation's execute function reads owner(), it's reading from the proxy's storage slot, not the implementation's. When it checks session keys, it's reading the proxy's session keys mapping. Each proxy is an independent wallet with its own state, sharing only the logic.
The implementation contract itself has its own storage too — the factory is its owner, and its initialization state is set to type(uint64).max by _disableInitializers. But nobody reads the implementation's storage in normal operation. It exists purely as a code repository that proxies delegate to.
This gives us a massive efficiency gain. Deploying a full MainAccount costs about 2.4 million gas. Deploying a proxy costs about 120,000 gas — that's a 95% reduction per wallet.


Slide 11: Testing and Deployment
"17 tests, 3 contracts verified on Sepolia"

The project has a comprehensive test suite with 17 tests covering all features. The tests use Foundry's local Anvil chain with a mock EntryPoint, and they validate everything from basic owner execution to the full UserOp flow through the EntryPoint, batch execution atomicity, session key creation and revocation and expiration, all four recovery scenarios, and factory idempotency and CREATE2 address prediction.
The complete system is deployed on Sepolia testnet: the MainAccountFactory, the MainAccount implementation, and my personal proxy wallet. I've successfully executed a full ERC-4337 UserOperation through the EntryPoint on-chain — the EntryPoint validated my signature, my proxy wallet transferred 0.001 ETH, and the bundler received gas compensation. All three contracts are verified on Etherscan so anyone can read the source code.
One challenge I encountered during deployment was gas estimation. Forge's default gas estimate was too low for the EntryPoint's internal gas accounting, causing transactions to revert with AA95 out of gas. Sepolia's extremely low gas prices — around 0.001 gwei compared to mainnet's typical 30-100 gwei — also required careful tuning of the UserOp gas parameters. These are the kind of practical issues you only discover through actual deployment.
